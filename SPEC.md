2026년 7월 전면 개정판(v2)에 대한 1차 개정(v2.1). 기존 "일기 아카이빙 + AI 심리상담" 기획을 대체한다.

**v2.1 개정 요지** (Fluidwave 벤치마크 및 AI 엔진 기술 검토 반영):
1. **컨텍스트 프로필** 신설 (3.4) — 모든 AI 호출에 사용자의 우선순위 판단 기준을 주입하는 핵심 장치
2. **하이브리드 우선순위 엔진** 명세 확정 (4.2) — LLM은 항목별 평가만, 최종 정렬은 코드가 결정론적으로 수행
3. **마일스톤** 도입 (3.3, 4.7) — 프로젝트와 할일의 중간 위상. `projects` 내부 배열로 구현, entries 스키마 불변
4. 정리 화면 기본 정렬·표시 명세 보완 (4.2)
5. 저녁 정리에 컨텍스트 프로필 원탭 갱신 단계 추가 (4.3), 포커스 카드에 "이번 주 상황" 표시 (4.4)
6. v2.1 이후 로드맵 신설 (11): 구글 캘린더 읽기 유입, 마인드맵 뷰, 습관(누적형)
7. 정합성 검토 반영: "정리하기"·포커스 대기열 대상을 할일(+기한 있는 프로젝트)로 한정, priority null 항목 배치 규칙, 프로필 갱신 노출 조건 초기값(null 포함), 인박스 분류 비동기 UX, 아카이브 로드 범위, 프로필 편집 UI의 개발 단계 분리
8. 사용자 질문지 기반 확정: 초기 컨텍스트 프로필 전문 (3.4), `#딥워크` 기능 태그 (3.2), 딥워크 요일 스케줄 (3.3, 4.2 — 코드가 판정, 설정에서 편집 가능), 데이터 경계·신뢰 모델 (6)
9. 최종 리뷰 반영: **표시 시점 오버레이** 신설 (4.2 — 딥워크·D-1·pinned 규칙을 "정리하기" 실행 여부와 무관하게 매일 보장, 서열: pinned > D-1 이하 > 딥워크 > priority), 근거 문구의 기한 언급은 마일스톤 포함 D-1 이하로 제한, 아이디어 리뷰에 active 프로젝트 목록 주입, status 'migrated' 제거 및 이월 동작 정의

## 1. 프로젝트 개요

ADHD 사용자의 작업기억을 보조하는 **브레인 덤프 + AI 트리아지** 웹앱.

- **사용 목적 (1)**: 떠오르는 아이디어를 마찰 없이 빠르게 덤프하고, 하던 과제에 즉시 복귀할 수 있게 한다.
- **사용 목적 (2)**: 수많은 아이디어 사이의 우선순위를 정리하고 실행으로 옮기는 것을 돕는다.
- 기존 일기 아카이빙·심리상담(카운슬러)·레퍼런스 DB 기능은 **부기능으로 축소 유지**한다.
- 개발자 1인(Claude Code) 단독 구현, 배포 대상 사용자 1인(ku.hayoungkim@gmail.com — Firebase 소유 계정과 동일. 개발자와 사용자는 다른 사람이다, 6장 신뢰 모델 참조). 모바일(갤럭시 Z 폴드 7) 중심, PC 및 다기기 접근 지원.

### 디자인 원칙

- **극도로 미니멀.** 화려한 연출은 인지부하를 늘리므로 배제한다. 완료 처리 시 취소선/페이드 등 최소한의 피드백만 사용.
- **필터링은 엄격하게, 아카이브는 직관적으로.** 실행 화면(포커스 모드)에서는 한 번에 하나만 보여주고, 보관된 데이터는 필요할 때 바로 찾을 수 있게 한다.
- **캡처는 3초 컷.** 앱 실행 → 입력 → 저장까지의 마찰을 최소화한다. 완벽하게 적지 않아도 일단 던져둘 수 있어야 한다.
- **죄책감 없는 이월.** 미완료 항목에 빨간 경고·지연 표시를 하지 않는다. 부드러운 톤으로 재검토를 유도한다.

### 아키텍처 원칙

- **Claude API·Vision API는 클라이언트에서 직접 호출하지 않는다.** 반드시 Vercel Serverless Function을 프록시로 경유한다. API 키는 서버 환경변수에만 존재하며, 함수 진입 시 Firebase ID 토큰을 검증해 허용된 이메일만 통과시킨다.
- **토큰 비용 절감 원칙**: 인박스 분류 제안은 저녁 정리 직전 일괄(batch) 1회 호출로 처리. 우선순위 재계산·할일 쪼개기·아이디어 리뷰·URL 요약·카운슬러 분석은 모두 사용자가 버튼을 눌러야만 실행.
- **컨텍스트 프로필 공통 주입 원칙**: 인박스 분류·우선순위 재계산·쪼개기·아이디어 리뷰 등 트리아지 계열 AI 호출의 시스템 프롬프트에는 항상 컨텍스트 프로필(3.4)을 주입한다. 프롬프트 조립은 서버리스 함수에서 수행한다.
- **LLM/코드 역할 분리 원칙**: LLM은 텍스트의 의미 판단(중요도·연관성·제안·근거 문구)만 담당하고, 산술(날짜 계산·D-day)과 최종 정렬은 코드가 결정론적으로 수행한다. 날짜 관련 근거 문구는 코드가 생성한다.

## 3. 데이터 구조

### 3.1 카테고리 (5종 + 인박스)

| 카테고리 | 정의 | 기한 | 이월/리뷰 |
|---|---|---|---|
| **프로젝트** | 장기 전략 구상 → 할일 체계화 → task 설계 → 실행 (이벤트성 행사 준비 포함) | 선택 | 하위 할일 단위로 이월 |
| **할일** | 1건짜리 실행 과업 | 선택 | O (죄책감 없는 이월) |
| **쇼핑** | 구매 목록. 태그 없음, 독립 리스트 | 없음 | 체크만 |
| **아이디어** | 스치는 생각. 실행과 직결되지 않으며 디벨롭 필요 | 없음 | 주간 리뷰 → 할일/프로젝트로 승격 또는 폐기 |
| **생각** | 순수 기록·일기·단상. 실행으로 이어질 필요 없음 | 없음 | 없음 (아카이브) |
| (인박스) | 미분류 상태. 카테고리·태그를 아무것도 선택하지 않고 저장한 항목 | — | 저녁 정리 시 일괄 처리 |

### 3.2 태그 체계 (2계층)

- **주제 태그** — `#교육` 형식. 고정 6종: `#일기` `#심리` `#교육` `#의료` `#일` `#금융`. 카운슬러·레퍼런스 부기능에서 사용.
- **기능 태그 (v2.1)** — `#딥워크` 1종. 우선순위 규칙을 발동시키는 태그: 이 태그가 달린 항목은 딥워크 요일에 최상위로 배치된다 (4.2, settings.deepWorkSchedule 참조). 태그 바에서 주제 태그와 같은 방식으로 부착하되, 시각적으로 구분(색상 등 최소한으로)한다.
- **프로젝트 태그** — `.논문작업` 형식. 프로젝트 생성 시 자동 발급. 할일·아이디어·생각을 프로젝트에 귀속시키는 수단 (별도 parentId 없음).
- 태그 바 UI: 프로젝트 태그 구간 / 주제 태그 구간을 나누어 표시. 최근 사용 태그만 노출하고 "더보기" 드롭다운으로 전체 접근. 프로젝트 태그 구간은 `projects` 컬렉션의 active 항목만 렌더링한다.

### 3.4 컨텍스트 프로필 (v2.1 신설 — AI 개인화의 핵심 장치)

AI가 활용할 수 있는 맥락은 시스템 안에 캡처된 텍스트가 전부다. 컨텍스트 프로필은 사용자의 우선순위 판단 기준을 AI에게 미리 알려주는 자유 텍스트로, 트리아지 계열 모든 AI 호출에 주입된다.

**3층 구조** (`settings.contextProfile`):

| 층 | 필드 | 갱신 주기 | 편집 위치 | 예시 |
|---|---|---|---|---|
| 불변 규칙 | fixedRules | 거의 불변 | 설정 화면 | "클라이언트 기한이 걸린 업무는 항상 최상위" |
| 분기 초점 | quarterFocus | 월~분기 | 설정 화면 | "7월: KMCC 명단 작업이 사이드 프로젝트 중 최우선" |
| 이번 주 상황 | weeklyStatus | 주 단위 | 저녁 정리 원탭 갱신 (4.3) | "수요일 클라이언트 미팅 준비가 모든 것에 우선" |

**작성 규칙** (설정 화면에 안내 문구로 노출):

- 가치 선언("가족이 중요")이 아닌 **조건 → 동작 형식의 판정 규칙**("아이 관련 항목은 기한 무관 상위 5위 안 배치")으로 작성해야 AI가 우선순위에 적용할 수 있다.
- 분량은 층별 5~7줄, 전체 20줄 이내. 규칙이 많아지면 상호 충돌한다.
- 규칙 간 충돌에 대비해 fixedRules 첫 줄에 우선순위 서열을 명시한다 (예: "충돌 시: 클라이언트 기한 > 가족 > 사이드 프로젝트").

초기 배포 시에는 아래 확정본으로 시드하며, 프로필이 비어 있어도 앱은 정상 동작해야 한다.

**초기 프로필 확정본 (사용자 질문지 기반 — 최초 배포 시 이 값으로 시드한다. 사용자는 설정에서 언제든 수정 가능):**

```
[불변 규칙 — fixedRules]
충돌 시 서열: 기한 임박(D-1) 할일 > 딥워크(딥워크 요일) > 일반 할일(기한순) > 아이디어 디벨롭
1. #딥워크 태그 항목은, 시스템이 오늘을 딥워크 요일로 알려준 경우 기한과 무관하게
   최상위로 배치한다.
2. 기한 있는 할일은 D-1에 최상위로 승격한다. 그 전에는 기한을 이유로 순위를 올리거나
   근거 문구에서 기한을 언급하지 않는다 — 장기 기한 추적은 피로를 유발함.
3. 미루는 항목(건너뛰기 3회 이상)에는 쪼개기를 먼저 제안하고, 쪼갠 뒤에도 실행되지
   않으면 폐기를 솔직하게 제안한다.
4. 폐기 제안은 과감하게 — 목록이 깔끔한 것이 우선. 단 확정은 항상 사용자가 한다.
5. 분류 기준: 실행에 옮기고 싶거나 옮겨야 하는 것 = 아이디어, 실행 없이 내면에
   머무르는 것 = 생각.
6. 아이디어의 할일·프로젝트 승격 제안은 보수적으로 — 산발적 확장보다 진행 중인 일의
   완수가 우선이다.

[분기 초점 — quarterFocus]
- 최우선 가치: 딥워크(논문 학습·작성, 법인 운영, 콘텐츠 제작·배포)의 습관화.
  딥워크 항목의 착수와 완수를 다른 무엇보다 우선한다.
- 새 일을 벌이기보다 이미 벌여놓은 할일의 완수를 우선한다.
- 항목 하나하나가 작은 성취가 되도록, 큰 덩어리에는 쪼개기를 적극 제안한다.
- 동시 진행 프로젝트는 3개를 넘기지 않는다 (승격 제안 시 준수).

[이번 주 상황 — weeklyStatus]
벌려놓은 할일 목록을 정리해 우선순위를 세우고 실제 실행으로 옮기는 주간 —
신규 확장 억제, 기존 항목 완수 우선.
```

## 4. 화면 구성

### 4.0 앱 전체 구조

- **하단 탭 3개**: [덤프] (캡처, 시작 화면) / [정리] (리스트·프로젝트·검색) / [포커스] (단일 카드 + 뽀모도로)
- **설정 하위 메뉴**: 카운슬러(부기능) / 레퍼런스(부기능) / 도파민 메뉴 관리 / 저녁 정리 시각 / 생체인증 / 사이드 버튼 설정 가이드

### 4.1 덤프 화면 (메인·시작 화면)

- 중앙: 깔끔한 텍스트 입력창 1개. 진입 즉시 커서 포커스. (음성 입력은 OS 키보드의 음성인식 기능 사용 — 별도 구현 없음)
- 좌측: 카테고리 아이콘 5종 세로 배치 (불렛저널 색인 스타일). 터치 시 해당 카테고리로 **즉시 확정 저장**.
- 우측: 기한 슬롯. 터치 시 달력 UI.
- 하단: 태그 바 (3.2 참조). 터치 시 태그 부착. 선택은 옵션.
- **아무것도 누르지 않고 저장** → `category: 'inbox'`로 저장. AI 분류는 즉시 실행하지 않고, 저녁 정리 직전 일괄 호출로 `suggestedCategory`를 채운다.
- [프로젝트] 버튼으로 저장한 경우 → 즉시 저장 후 **프로젝트 상세 설정 화면으로 자동 전환** (이름 확정, `.태그` 자동 발급, 목표·기한 입력).

### 4.2 정리 화면

- 카테고리별 필터, 텍스트 검색 (클라이언트 사이드 필터링 — Firestore에는 전문검색이 없으므로 로드된 항목을 프론트에서 검색).
- 인박스 badge 표시.

**기본 표시·정렬 명세 (v2.1 보완)**:

- 기본값: 미완료(open)만 표시, 카테고리 전체. "정리하기" 실행 이력이 있으면 priority순, 없으면 최신순.
- priority가 있는 항목은 순번과 AI 근거 한 줄을 함께 표시 — 이 목록이 곧 포커스 모드의 대기열이며, 사용자가 AI 정렬을 검수하는 화면이다.
- **priority가 null인 항목** (마지막 "정리하기" 이후 캡처된 할일 등): priority 보유 항목들 아래에 최신순으로 배치하고 "미정렬" 시각 표시(옅은 점 등 최소한으로). 다음 "정리하기" 실행 시 대기열에 편입된다. 급한 신규 항목은 pinned로 수동 편입 가능.
- 카테고리 필터 선택 시 정렬 자동 전환: 할일·프로젝트 = priority → 기한순 / 아이디어·생각 = 최신순 / 쇼핑 = 입력순.
- 상단에 완료/보관 토글 1개 (기본 off). 아카이브 접근은 이 토글 + 텍스트 검색으로 해결한다. **로드 범위**: done/archived/discarded 항목은 기본 로드하지 않고, 토글 on 시점에 최근순 페이지네이션으로 추가 로드한다 (데이터 누적 시 전체 로드 비용 방지). 검색은 로드된 범위 내에서 동작함을 전제로 한다.

**"정리하기" 버튼 — 하이브리드 우선순위 엔진 (v2.1 명세 확정)**:

**평가 대상 = 미완료 할일 + 기한 있는 프로젝트로 한정한다.** 쇼핑(체크만)·아이디어(주간 리뷰가 전담)·생각(아카이브)은 카테고리 정의(3.1)상 우선순위 개념이 없으므로 평가에서 제외한다 — LLM 입력 토큰 절감 효과도 있다. 같은 이유로 **포커스 모드 대기열도 할일만**으로 구성한다.

순수 LLM 랭킹은 실행마다 순위가 출렁이고(비일관성), 날짜 셈을 틀리며(근거 환각), 목록 중간 항목을 소홀히 평가한다(위치 편향). 따라서 다음과 같이 역할을 분리한다.

1. **LLM (Claude API) — 항목별 구조화 평가만 수행.** 입력: 평가 대상 항목(위 한정 기준) + 컨텍스트 프로필(3.4) + active 프로젝트들의 goal 및 미완료 마일스톤 목록. 출력(항목별 JSON): `{중요도: 1~5, 목표연관: 기여하는 프로젝트/마일스톤, 제안: 쪼개기/폐기 여부, 근거 문구}`. 전체 순위는 출력하지 않는다.
2. **코드 (서버리스 함수) — 결정론적 최종 정렬.** 기한 임박도(코드가 날짜 계산) × 중요도 × 목표연관을 가중 합산. pinned 항목 순위 보호. 직전 순위 대비 변경 최소화(안정성 앵커) 적용. 날짜 관련 근거 문구("D-2")는 코드가 생성하고, 의미 판단 근거만 LLM 문구를 사용한다.

**딥워크 요일 판정 (v2.1 — 코드 담당):** 코드가 `settings.deepWorkSchedule`과 오늘 요일로 `isDeepWorkDay`(해당 요일 시간 > 0)를 산출한다. LLM 프롬프트에도 주입하되, 실제 배치 보장은 아래 표시 시점 오버레이가 담당한다. AI가 요일을 추측하게 하지 않는다.

**표시 시점 오버레이 (v2.1 — 날짜 의존 규칙의 이원화):** 저장된 priority는 "정리하기" 실행 시에만 갱신되므로, 버튼을 누르지 않은 날에는 그날의 딥워크·기한 규칙이 반영되지 않는 구멍이 생긴다. 따라서 날짜 의존 규칙은 저장값이 아니라 **포커스 대기열·정리 탭을 로드할 때마다 클라이언트가 "오늘" 기준으로 결정론적으로 재적용**한다 (LLM 호출 없음, 비용 0). 오버레이 서열:

1. **pinned** — 기존 위치 유지 (최우선 보호)
2. **기한 D-1 이하** (당일·경과 포함) — 최상위 승격. 기한 경과 항목은 저녁 이월 리뷰가 처리할 때까지 이 위치를 유지한다
3. **딥워크일의 `#딥워크` 항목** — 그 다음 상위
4. **저장된 priority 순** — 나머지

이 서열은 프로필의 충돌 서열(기한 임박 > 딥워크)과 일치하며, 코드 명세로서 프로필 문구보다 우선한다.

- skipCount 3회 이상 항목은 쪼개기/폐기 제안. migratedCount 3회 초과 항목도 쪼개기/폐기 제안.
- **근거 문구의 기한 언급 규칙**: 할일 기한이든 마일스톤 기한이든, 근거 문구에서 기한은 D-1 이하일 때만 언급한다 (장기 기한 노출 = 피로원이라는 프로필 규칙 2의 취지를 마일스톤에도 동일 적용).
- 마일스톤 연관 근거 예시: ".논문작업의 '초고 완성' 마일스톤에 직결되어 올렸어요." (기한 언급 없음)

### 4.3 저녁 정리 (통합 리뷰 화면)

푸시 알림 없이 **로컬 트리거**로 동작: 설정된 시각(기본 20시) 이후 앱을 열면 처리할 항목이 있을 경우 이 화면을 우선 노출.

1. **인박스 정리**: 각 항목에 AI 제안 라벨 표시 → 원탭 확정 또는 수정. **제안 도착 전에도 수동 분류는 즉시 가능** — batch 호출의 응답을 기다리는 로딩 화면을 두지 않고, 제안 라벨은 도착하는 대로 비동기로 채워 넣는다 (체감 지연 제거).
2. **할일 이월 리뷰**: 대상 = (a) 기한 지난 할일 + (b) 포커스 모드에 노출됐으나 미완료인 항목. 각 항목에 `>` (내일로) / `>>` (주말로) / 완료 / 폐기 선택지. 빨간 경고 금지. "내일 하면 되죠 — 오늘 이만큼 적어둔 것도 진전이에요" 톤의 카피 사용.
3. **아이디어 리뷰** (주간): 마지막 리뷰로부터 7일 경과 시 이 화면에 섹션으로 함께 노출. AI가 항목별로 할일 전환 / 프로젝트 승격 / 폐기 / 보류를 제안 → 사용자 확정. 승격 시 category 변경, 프로젝트 승격 시 프로젝트 상세 설정 흐름으로 연결. **입력에 active 프로젝트 목록(이름·goal)을 주입한다** — 프로필의 "동시 진행 프로젝트 3개 초과 금지" 규칙을 AI가 판단하려면 현재 개수를 알아야 한다.
4. **컨텍스트 프로필 갱신 (v2.1)**: 마지막 단계에 현재 "이번 주 상황"(weeklyStatus) 텍스트를 표시하고 **[그대로 두기] / [고치기]** 원탭 선택. 매일 입력을 강요하지 않는다 — 상황이 바뀐 주에만 고치면 된다. 노출 조건: `weeklyStatusUpdatedAt`이 **null이거나 7일 초과**인 경우에만 노출해 리추얼 비대화를 방지한다 (최초 사용 시 null이므로 첫 저녁 정리에서 노출됨). 갱신하면 이후 모든 트리아지 AI 호출에 즉시 반영된다.

### 4.4 포커스 모드

- 우선순위 1위 항목 **단 하나만** 큰 카드로 표시. **대기열은 할일 카테고리만**으로 구성한다 (4.2 평가 대상 한정과 동일 기준 — 쇼핑·아이디어·생각은 카드로 노출하지 않는다).
- 카드 상단에 컨텍스트 프로필의 "이번 주 상황"(weeklyStatus) 한 줄을 작은 텍스트로 표시 (v2.1) — 포커스 이탈 시 "내가 지금 왜 이걸 하지?"의 닻 역할. 별도 입력·데이터 없이 기존 프로필을 재사용한다. 비어 있으면 표시하지 않는다.
- 버튼 3개: **[완료]** / **[건너뛰기]** (다음 순위 카드 표시, skipCount +1, 죄책감 유발 문구 없음) / **[쪼개기]** (4.6 직결).
- 카드에서 바로 뽀모도로 타이머 시작 가능.

### 4.5 뽀모도로 타이머 (부가기능)

- 고정 패턴: **25분 집중 – 5분 휴식 – 25분 – 5분 – 25분 – 15분 긴 휴식**.
- 숫자 카운트다운이 아닌 **비주얼 타이머** (색 영역이 줄어드는 Time Timer 형태) — 시간감각(Time Blindness) 보완.
- 휴식 시작 시 도파민 메뉴에서 랜덤 1개 추천 노출 (설정에서 사용자가 목록 등록).

### 4.6 할일 쪼개기 마법사

- 할일 상세 및 포커스 카드에서 [쪼개기] 버튼 → Claude API 호출 → "시작할 수 있는 최소 단위" 3~4단계 하위 체크리스트 생성 → 사용자 확인 후 저장.

### 4.7 프로젝트 상세

- 해당 `.프로젝트태그`가 달린 모든 항목(할일/아이디어/생각) 모아보기 + 프로젝트 목표·기한·상태 관리.
- 프로젝트를 done/paused로 변경하면 태그 바에서 자동 제외.

**마일스톤 관리 (v2.1)**:

- 프로젝트 상세에서 마일스톤 목록 표시·추가·수정·삭제·순서 변경. 완료는 **수동 체크** — 하위 할일 완료율로 자동 계산하지 않는다 (자동 계산에는 할일–마일스톤 구조적 연결이 필요해지므로, 평평한 데이터 구조를 지키는 대가로 수용).
- 할일과 마일스톤은 **구조적으로 연결하지 않는다.** entries에 milestoneId 등 참조 필드를 추가하지 않는다. 연결은 "정리하기" 시 AI가 goal·마일스톤 컨텍스트를 바탕으로 추론한다 (4.2).
- **[마일스톤 제안] 버튼**: Claude API 호출 → goal과 기존 할일들을 바탕으로 마일스톤 3~5개 제안 → 사용자 확인 후 저장. 쪼개기 마법사(4.6)가 아래 방향 분해(할일 → 최소 단위)라면, 이 기능은 위 방향 분해(목표 → 세부 목표)로 대칭 구조를 이룬다. 버튼 클릭 시에만 호출(비용 원칙 준수).

### 4.8 카운슬러 / 레퍼런스 (부기능, 축소)

- 기존 기획의 기능을 유지하되 하단 탭에서 제외하고 설정 하위로 이동.
- 카운슬러: 기간·태그 선택 → 분석 실행(버튼 클릭 시에만 API 호출) → 현재 상태 요약 / 조언 / 액션플랜 / 생각해볼 주제 + 후속 채팅. 심리 마스터 요약본(psych_summary/master) 생성·갱신 버튼 포함.
- 레퍼런스: 사진 OCR(원본 미보관) / 유튜브 URL(자막 추출 후 요약) / 기사 URL(본문 크롤링 후 요약). 요약은 버튼 클릭 시에만.

## 5. 오프라인 및 빠른 실행

- **오프라인 캡처 보장**: Firestore 오프라인 persistence 활성화. 오프라인 저장은 로컬에 즉시 기록, 온라인 복귀 시 자동 동기화. AI 분류 제안은 동기화 이후 처리. 덤프 화면은 Service Worker로 프리캐시하여 비행기 모드에서도 열린다.
- **PWA shortcut**: Manifest `shortcuts`에 "새 덤프" 등록 — 홈 화면 아이콘 롱프레스로 덤프 화면 직행(커서 포커스 상태).
- **사이드 버튼 빠른실행 (기기 설정 안내)**: 갤럭시 Z 폴드 7 기준, 설정 > 유용한 기능 > 사이드 버튼 > 두 번 누르기 > 앱 실행에서 설치된 PWA 지정 (One UI 버전에 따라 메뉴명 상이할 수 있음). 설치 가이드 문서에 포함.

| 생체인증 | WebAuthn. 기본 비활성화, 설정에서 on/off |

### 데이터 경계·신뢰 모델 (v2.1 확정)

본 앱은 개발자(gomtang)와 사용자(ku.hayoungkim)가 다른 **가족 간 신뢰 기반 1인용 앱**이다. 격리의 목적은 프라이버시가 아니라 **상호 데이터 오염 방지**다.

- **완전 독립 원칙**: 연어항해일지의 모든 데이터(entries, projects, settings, 심리 기록)는 사용자 소유 Firebase 프로젝트 내부로만 통제된다. 개발자의 업무 데이터·저장소·설정을 참조하지 않으며, 역방향 유입도 없다. 개발은 Family 저장소에서 독자적으로 수행하고, 프로젝트 관련 개발 지침은 해당 저장소의 CLAUDE.md에 자체 완결적으로 둔다 (전역 설정 의존 금지).
- **신뢰 범위 결정사항**: Claude API·Vision API 키는 개발자 계정 것을 사용한다 (토큰 결제 주체 = 개발자). 개발자와 사용자는 상호 데이터 열람이 무방하다 — 통제 대상은 외부 노출뿐이다.
- **개발 데이터 원칙**: 개발·디버깅은 시드(가짜) 데이터로만 수행한다. 실사용 데이터 입력은 배포 후 사용자 본인만 한다. 개발 단계에서는 Firebase 에뮬레이터 또는 별도 테스트 프로젝트를 사용해 운영 Firestore를 건드리지 않는다.

## 11. v2.1 이후 로드맵 (본 개발 완료 후 착수)

| 순위 | 기능 | 요지 |
|---|---|---|
| 1 | **구글 캘린더 읽기 전용 유입** | 분업 규칙: "시각이 정해져 있으면 캘린더, 아니면 덤프" — 캘린더가 time-sensitive 항목의 단일 진실 소스. "정리하기"·저녁 정리 시 향후 7~14일 일정을 읽어 프롬프트에 주입 (컨텍스트 프로필의 자동화된 4번째 층). 캘린더에 쓰기 없음. 기술: Firebase Google Auth에 calendar.readonly 스코프 추가, GCP 테스트 모드(1인 사용, 앱 심사 불요), 액세스 토큰 1시간 만료 → 갱신 흐름 필요. 선택 검토: 기한+시각이 지정된 할일의 "캘린더에 넣기" 원탭 단방향 생성 버튼 |
| 2 | **마인드맵 뷰 (전체 조망)** | 가로형(좌→우) 트리: 프로젝트 → (마일스톤 가지 ∥ 할일 가지 — 구조적 연결이 없으므로 나란히 배치) + 미분류 클러스터. 읽기 전용 투영이므로 스키마 변경 없음. 기본은 프로젝트 레벨만 펼침, 탭으로 가지 접기/펼치기, 핀치 줌 + 팬 (Z 폴드 7 펼침 화면 최적화). 노드 신호는 기한 임박·pinned·skipCount 누적만 최소 표시. 기술 후보: React Flow + 자동 레이아웃 |
| 3 | **습관 (누적형 반복 할일)** | `recurrence` 필드(daily/weekly). 스트릭(연속)이 아닌 **누적 횟수** 표시("이번 달 12회") — 끊긴 스트릭은 죄책감 장치이므로 배제. 미실행일은 이월 리뷰에 등장시키지 않고 다음 주기에 조용히 재등장 |

## 12. 참고사항

- Claude API 모델명은 소스 내 설정 파일 한 곳에서만 관리한다 (기본: claude-sonnet-4-6).
- 심리 마스터 요약본은 최초 1회 생성 필수, 이후 #심리 기록 누적 시 수동 갱신 (기존 기획 유지).
- 네이버 블로그 아카이빙 데이터는 별도 정규화 스크립트로 Firestore에 일괄 업로드 예정 (본 개발과 별개 작업, 기존 기획 유지).
